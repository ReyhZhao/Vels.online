# A Rule Test evaluates a Scheduled Search Rule against an ephemeral real OpenSearch index, not a simulator

A **Scheduled Search Rule** ([ADR-0006](0006-scheduled-search-rules-pull-engine.md)) is authored, activated, and then trusted to keep firing correctly as its legs, the Wazuh schema, and OpenSearch behaviour all drift over time. We want detection-as-code testing for these rules: ship a rule with **Rule Tests** — saved bundles of synthetic **Sample Documents** plus a fire / no-fire **Expectation** — that an author can run on demand to prove the rule still behaves as intended. The load-bearing decision is *how* a Rule Test evaluates the rule: we run the **real** production matcher (the same compiler, key-join, and Diversity-Constraint logic) against an **ephemeral OpenSearch index** that the testing flow creates, mapping-clones from the live alerts index, bulk-loads with the Sample Documents (pushed straight into OpenSearch, bypassing the Wazuh pipeline), queries, and drops — rather than reimplementing OpenSearch semantics in Python. To make this safe, the evaluator is split into a side-effect-free **decide** phase and a **materialise** phase, and the test path runs decide-only.

## Considered Options

- **Real engine against an ephemeral real OpenSearch index (chosen)** — the test exercises the exact code production runs (`compile_query` / `compile_agg_query`, the per-leg `terms` agg + Python key-join, the diversity cardinality check). A green test therefore means the *real* matcher works, which is the entire point of a regression guarantee. Costs: OpenSearch must be reachable to run a test, the client needs write methods it does not have today, and an index lifecycle must be managed.
- **In-memory Python simulation of OpenSearch** — evaluate the compiled DSL against the Sample Documents locally (term vs match vs wildcard, `.keyword`, range, CIDR, `terms` aggregation, distinct-cardinality). Fast, deterministic, no external dependency — but it is a *second, divergent matcher*. The day it disagrees with real OpenSearch (analyzer behaviour, multi-field resolution, type coercion *will* eventually disagree) a Rule Test goes green while the live rule is broken — the exact failure mode the feature exists to prevent. Rejected: it tests a lie.
- **Run against the real `wazuh-alerts-*` indices** (e.g. tag and later delete test docs) — rejected: writing synthetic documents into production indices pollutes dashboards and other rules' queries, and anything matching the `wazuh-alerts-*` glob is visible to every consumer of that glob.
- **Let dynamic mapping type the test index** instead of cloning the live mapping — rejected: OpenSearch query behaviour is governed by the mapping, so a dynamically-inferred index silently drifts from production (an `ip` field inferred as `text`, a missing `.keyword` subfield) and quietly destroys the fidelity that justified choosing the real engine in the first place.

## Consequences

- **The test index is ephemeral, isolated, and faithfully mapped.** Each run creates a uniquely-named, **non-glob-matching** index (e.g. `vels-ruletest-<runid>`), **clones the live alerts-index `_mapping`** onto it, bulk-loads the Sample Documents, `refresh`es so they are immediately searchable, runs, and **drops the index in a `finally`**. It must never be named to match `wazuh-alerts-*`. This requires new write capability on the otherwise read-only `OpenSearchClient` (create index, bulk index, refresh, delete).
- **Documents are pushed directly into OpenSearch, bypassing Wazuh.** A Rule Test does not route synthetic logs through the Wazuh→indexer pipeline; the testing flow owns index creation and indexing itself. Sample Documents may be *partial* raw Wazuh docs — only the window timestamp field and every field the rule references (condition / correlation-key / diversity) need be present.
- **The evaluator splits into `decide` and `materialise`.** The production path (`run` → `_run_*_leg` → `_materialise_and_fire`) currently interleaves the firing *decision* with its *effects* (real `Alert`, `Incident`, `SearchFiring`, `SearchFinding` rows). The test path reuses the decision and **performs zero DB writes and creates no Incident** — otherwise testing a rule would page an analyst and pollute production state. The refactor also makes the production evaluator more unit-testable in the repo's own pytest suite. A future reader should treat decide/materialise as a deliberate seam, not accidental layering.
- **Tenant scope is neutralised; the Window is preserved but sample-anchored.** The test runs with `agent_ids=None` (the existing "omit agent filter" path), because the test index contains only the test's own docs and per-rule tenant isolation is identical plumbing not worth re-asserting per rule. The **Window** *is* detection logic for multi-leg co-occurrence, so it is kept — but "now" is anchored to the latest Sample Document timestamp rather than wall-clock, so a test written today still passes next year.
- **The Expectation is whole-rule and boolean.** A Rule Test asserts the rule fires (≥1 satisfied key) or does not — never a **Leg** or **Condition** in isolation, and not finding-count or which-key. The **Test Result** surfaces rich diagnostics (satisfied key, per-leg hit counts, diversity found-vs-required) for *explaining* a failure, but those are not authored assertions. This keeps tests black-box and robust to rule refactors that preserve behaviour.
- **Runs are synchronous and bounded.** A test run (create → load → decide → drop) happens in-request, like the existing read-path OpenSearch calls, giving instant feedback; a per-test sample cap and the client's existing timeout bound the blast radius. A slow/unreachable OpenSearch surfaces as a test error, not a hang.
- **Authoring reuses the existing grounded assistant.** Sample Documents are hand-written or drafted by the LLM generator (TP + TN candidates) grounded in the live mapping and real sample docs ([ADR-0005](0005-rule-author-assistant-ephemeral-stateless.md), [ADR-0007](0007-search-rule-dynamic-field-catalog.md)), always human-reviewed before save and never auto-run. Capturing samples from real **Finding**s ("live import") is deferred to v2 — it drags in PII-in-fixtures and a chicken-and-egg problem for never-fired rules.
- **Tests are staff-only and tenancy-flat.** Rule Tests inherit `_require_staff` like every other `SearchRule` endpoint. A System Rule (org = null) has one set of Rule Tests, not one per tenant; because execution runs against synthetic data with scope neutralised, there is no per-org fan-out and **Mute** does not interact with testing.
- **Deferred:** scheduled/automatic test runs (v1 is on-demand only), run history (rely on the last-result summary stored on the test), capture-from-real Sample Documents, and richer Expectations (finding-count / expected-key / leg-level).
