# Incident Presence is a per-incident ephemeral registry streamed over SSE, with a soft comment-edit lock

We want analysts working an **Incident** to see, in real time, **who else is on it and what each is doing** — passively *viewing*, *working* a specific **Task**, or *editing* a specific **Comment** — and the incident's **AI agents** (a **Triage Agent** re-run, the **Incident Assistant**) the same way ([CONTEXT.md](../../CONTEXT.md) → Incident Presence). The goal is to stop analysts unknowingly colliding on the same incident, the same task, or — destructively — the same comment body. This is the first piece of *collaboration* presence in the app, and [ADR-0027](0027-live-attack-map-presence-gated-shared-snapshot-over-sse.md) anticipated it ("if live alert/incident streams follow, they should reuse this producer/shared-buffer/presence shape") — this ADR records that, on inspection, the incident case fits a **different** shape and says why.

## Decision

**A third SSE shape: the registry is both the source of truth and the thing streamed.** Each open incident detail view holds one long-lived SSE connection (`GET /api/incidents/<display_id>/presence/`) that polls a **per-incident presence registry in the cache (Redis)** every ~1–2s and emits the current **roster** whenever it changes. There is **no producer, no shared buffer, and no Celery beat** — unlike the Live Attack Map, whose presence registry exists only to *gate an expensive OpenSearch producer*. Here the data being streamed *is the roster itself*; there is nothing to gate and nothing to produce, so that machinery collapses away. And unlike `HuntEvent`, the roster is deliberately ephemeral, so it lives in the cache, not Postgres — no migration, no prune job, no write-DB churn for state that is meaningless in 15 seconds.

**Declared activity rides an inbound POST; viewing is passive.** SSE is one-way, so *viewing* is inferred server-side from the open stream (the GET loop heartbeats the actor each tick, exactly as `attackmap/presence.py` beats `conn_id`), while `working`/`editing` transitions the client cannot be inferred are pushed up via `POST /api/incidents/<display_id>/presence/ {activity, target}`. Records are keyed **`(incident, actor)`** (not per-connection — the roster answers "is Dana here," tabs collapse, last-write-wins), carry `{activity, target, expiry}`, and use a ~15s TTL refreshed by heartbeat so a dead tab self-heals. The heartbeat refreshes *expiry only* and preserves `activity`, so the passive viewing beat never clobbers a declared `editing`/`working` state. The roster is a **full snapshot** — no `?after=`/seq resume (there is no event history to replay, unlike Hunt and the attack map).

**Tasks are advisory; editing an existing Comment is a soft lock — a deliberate asymmetry.** `working` a Task is *focus-derived and advisory*: it is set when the analyst expands/focuses a task, never mutates durable `Task.state`/`assignee`, and two analysts may focus the same task (Presence informs collision, it does not prevent it). `editing` an *existing* Comment is *exclusive*, because `Comment.body` is last-write-wins and genuinely clobberable: the first editor holds a lock; others get a read-only editor attributed to the holder. The lock is not a separate store — it **rides the same presence record and TTL**, so a dead tab releases it for free — plus an **idle auto-release** (~5 min since the holder's last keystroke, refreshed on typing) so a walked-away-but-open editor cannot deadlock the comment. Composing a *new* Comment is never locked. The asymmetry is intentional: the lock exists exactly where there is a concrete data-loss risk and nowhere else; do not "fix" the inconsistency by locking tasks or un-locking comments.

**AI actors are first-class roster members, sourced from the orchestrator, not a connection.** Humans announce presence by holding the stream and heartbeating; AI agents run server-side and hold no browser connection, so their presence is written into the same registry from the shared agentic orchestrator's step callbacks (the [ADR-0014](0014-incident-assistant-sse-streaming.md) `on_event` boundaries) and dropped in the run's `finally`. An AI actor is a *synthetic* roster entry (`actor_kind=ai`, e.g. "🤖 Incident Assistant"), not a `User`; the Assistant is **attributed to its invoking analyst**. AI actors never contend for the comment lock because they only ever *create* new comments (a Triage re-run appends a fresh `ai_triage` comment; it never mutates an existing body).

**Fail open, degrade invisible.** The incident detail view is a critical SOC surface; Presence is a non-critical overlay. If the cache is unavailable, presence reads/writes silently no-op (no roster, no errors, the page is exactly as it is today). The soft lock **fails open**: if the lock cannot be checked or acquired, the client allows the edit anyway, degrading to today's unlocked behaviour. A rare lost update under a cache outage is a far smaller harm than blocking all comment editing because a cosmetic dependency is down.

## Considered Options

- **Third-shape registry-as-source-and-stream (chosen)** — no producer/buffer/beat to operate; ephemeral state lives in the cache where it belongs; the lock self-heals off the presence TTL for free. Cost: a new long-lived connection profile and a small inbound POST path.
- **Reuse the Live Attack Map producer/shared-buffer/presence shape (rejected)** — the literal forward-looking suggestion in ADR-0027, but that shape's whole point is a presence *gate* in front of an expensive shared producer. Incident Presence has no expensive producer and no shared snapshot to compute — the roster is self-describing — so the producer, buffer, and beat would be dead machinery.
- **Persist presence like `HuntEvent` in Postgres (rejected)** — maximal consistency with the Hunt pattern and trivially queryable, but it is durable storage for deliberately throwaway data: a migration, a prune job, and write-DB churn every heartbeat. The cache fits ephemeral, auto-expiring state.
- **A standalone comment-lock table with its own ownership/expiry/steal protocol (rejected)** — explicit and queryable, but it duplicates a liveness/expiry mechanism the presence record already has; building the lock *on* the presence TTL means a dropped socket releases the lock with zero extra code.
- **Advisory-only comments, no lock (rejected)** — consistent with the task stance and cheaper, but it leaves the real last-write-wins clobber on `Comment.body` unaddressed; the soft lock was chosen specifically for that data-loss risk.
- **AI actors excluded from the roster (rejected)** — tempting because the *first automatic* Triage run fires before any human is watching, but re-runs and Assistant actions happen *while analysts watch*, and surfacing that live AI work is exactly the point.

## Consequences

- **A new long-lived connection profile.** Unlike every prior SSE surface (short-lived assistant turns; deliberately-opened staff dashboards), this holds a connection for the whole time a page is open. It is kept cheap by three constraints: the poll loop is **DB-free** (the roster carries actor identity in the cache record — never an ORM hit per tick, despite ADR-0014's note on held connections multiplying Postgres connections), the client **tears the stream down on tab-hidden** (Page Visibility) so connections track *actively-looked-at* tabs and a backgrounded tab both drops off the roster and releases any held lock, and the audience is bounded (staff-only, ≤ ~20 staff).
- **This amends ADR-0027's forward note.** That ADR's "reuse this shape for live incident streams" should be read as "reuse the *cache-presence + per-connection SSE tail* primitives," **not** the producer/shared-buffer/gate topology, which applies only when an expensive shared producer needs gating.
- **The wire contract is a client protocol, not just an endpoint** — a roster snapshot shape plus the `{activity, target}` POST and its 409-on-lock-held response — so changing the roster/activity shape is a coordinated server+client change.
- **The lock is advisory-grade, not a guarantee.** It fails open by design and self-releases on TTL/idle, so it prevents the *common* accidental clobber, not a determined concurrent overwrite. If hard last-write-wins protection on `Comment.body` is later wanted, that is optimistic concurrency on the comment PATCH endpoint (an `updated_at` version check) — a separate, orthogonal data-integrity change, not part of the presence channel.
- **Staff-only in v1.** Customer org members can access an incident but neither see nor emit presence; widening to a cross-audience roster is a later change the actor-keyed record already accommodates.
