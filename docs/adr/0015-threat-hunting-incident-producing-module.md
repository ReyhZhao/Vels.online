# Threat Hunting is a distinct, persisted, incident-producing module reusing the assistant plumbing

The **Hunt** module lets a SOC staff member run an LLM-assisted threat hunt — seeded by a free-text question or a link to an external malware/threat report — that ranges over Wazuh telemetry (both IOC sweeps and behavioral pattern hunting) and surfaces what needs investigation. It **reuses** the Incident Assistant's plumbing (the ADR-0011 agentic orchestrator, providers, native web search, and the ADR-0014 SSE event vocabulary) but is its **own concept**, because the Incident Assistant is incident-*bound* (lives inside one incident, seeded and grounded by it, ephemeral) while a Hunt is incident-*producing* (starts from no incident and creates them). v1 is **staff-only and cross-org** (the SOC hunts across all tenants — a sanctioned staff right per ADR-0011); v2 may open constrained single-org hunting to org members (tracked separately).

## Decision

- **A Hunt is a first-class, persisted, resumable, auditable entity** — seed, owner, org scope, lookback window, transcript/tool-trace, Findings, and links to spawned incidents. This is a deliberate divergence from the Incident Assistant's *ephemeral* conversation (ADR-0008): a Hunt is long-running and expensive, a staff member must be able to continue it later and review past hunts, and a record of *who hunted what across which tenants and what they found* is the audit trail for cross-tenant access. A Hunt's durable transcript **is** its grounding (there is no incident seed to recompute server-side each turn).
- **Reuse our own org-scoped clients; reject the external Wazuh MCP server (`gbrigandi/mcp-server-wazuh`).** Hunt tools wrap the existing `OpenSearchClient`/`WazuhClient`, which are already org-scoped and already reuse our authorization (ADR-0011's "never a backdoor"). The external MCP server is rejected because it (a) exposes **no arbitrary indexer query** — so it cannot do the core IOC sweep or behavioral aggregation — (b) is **global/manager-wide with no org→agent scoping**, violating the tenant-isolation invariant the whole codebase is built on, and (c) duplicates, more coarsely, capabilities we already have. The one thing it adds (live process/port enumeration) is a thin Wazuh Manager API call `WazuhClient` can make directly.
- **Composable fixed-lens tools in v1** (IOC-search-by-hash/ip/domain plus behavioral primitives: top rules, event histogram, top values, agent activity, processes/ports), all sharing the org/window scoping. Chosen over one general restricted-DSL query tool to stay inside ADR-0011's reliability guidance for the Ollama runtime and keep aggregation cost/abuse bounded and testable. The guarded general-query escape hatch is deferred to v2 (issue #473).
- **Propose-and-confirm output, partitioned per org — never auto-create.** A Hunt surfaces **Findings grouped by affected org**; each group becomes **one proposed Incident** the staff member confirms with a click, at which point the matched raw Wazuh docs are **materialised** as Alerts (reusing the Scheduled Search Rule bridge) and linked to the new Incident in *that org's* scope. This keeps ADR-0004's human-in-the-loop principle intact for the most lifecycle-affecting act in the system, and tenant isolation by construction: cross-org means the hunt *reads* widely but **never** produces a cross-tenant joined incident. The Hunt auto-executes only internal, reversible artifacts (its own notes/findings), per ADR-0012's risk grading.
- **Cross-org scope is explicit and recorded, defaulting to all orgs**, narrowable to selected orgs/agent groups, with a staff-selectable lookback window. Even an "all orgs" hunt **fans out per tenant** at query time and a lens never joins across tenants — mirroring the System Search Rule fan-out invariant.
- **Egress.** Web search is **unrestricted** (generic threat research, not bound to any one customer's disclosure rules — matching the rule-author assistant precedent in ADR-0011; the incident PAP gate has nothing to bind to since a Hunt has no incident). The staff-supplied **report-URL fetch runs server-side behind a hardened SSRF guard** (scheme allowlist, reject private/link-local/loopback/metadata ranges, cap size, no blind cross-host redirects) and its content is treated as untrusted, prompt-injection-aware input.

## Considered Options

- **Extend the incident-bound assistant instead** — rejected; a Hunt has no incident to bind to and is incident-producing, a structurally different grounding/authority/surface.
- **Use the external Wazuh MCP server** — rejected; lacks the core query tools, breaks tenant isolation, redundant (see Decision).
- **Auto-create incidents on a finding** — rejected; reverses ADR-0004 for the highest-consequence act, and the cross-org multiplier (one seed → N tenants → N incidents) makes silent creation worse, not better.
- **Ephemeral like the Incident Assistant** — rejected; defeats resume, audit, and the reconnectable streaming a long cross-org sweep needs (see ADR-0016).
- **One general restricted-DSL query tool in v1** — deferred to #473; too large a schema for the primary runtime to drive reliably without real usage data to bound its allowlist.

## Consequences

- A new persisted `Hunt` aggregate (+ Findings + spawned-incident links) and a staff "Threat Hunting" console, separate from any incident view.
- The materialise bridge, previously Scheduled-Search-Rule-only, gains a second caller; `Finding` is widened in CONTEXT.md to cover a Hunt's matched docs.
- Execution model diverges from ADR-0014 — see [ADR-0016](0016-hunt-runs-as-background-job-streamed-over-sse.md).
- v2 opening to org members reopens the egress and scope defaults (own-org-only, re-gated web search) — tracked separately.
