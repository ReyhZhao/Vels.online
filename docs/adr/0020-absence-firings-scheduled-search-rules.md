# Absence Firings: Scheduled Search Rules can trigger on the *absence* of matching documents

[ADR-0006](0006-scheduled-search-rules-pull-engine.md) framed a **Scheduled Search Rule** entirely around presence: a scheduled query finds matching documents, **Materialise**s them as in-app **Alert**s (**Findings**), and produces one **Incident** whose evidence *is* those Alerts. Some detections are the inverse — the signal is that something expected *stopped happening*. The motivating case: "no firewall logs received in the past hour" as an alert that the firewall logging went silent (a dead-man's-switch).

We make the per-leg document-count threshold operator configurable. Each **Leg** gains a **Count Operator**: `gte` (default, the existing "at least N matched" behaviour) or `lte` (the inverse). A leg with `lte` triggers an **Absence Firing** when too few — typically zero — documents matched within the **Window**.

## Considered Options

- **Zero-Alert Incident, evidence in the description (chosen)** — an Absence Firing creates an Incident directly with **no Findings and no Alerts**; the shortfall itself (window, expected vs actual count, the condition that was absent) is recorded in the Incident description and a `SearchFiring` ledger row. Honest — it fabricates no data — and it reuses the existing Incident / IOC / triage pipeline, none of which structurally require linked Alerts (Incident creation is independent of Alerts; IOC extraction reads the Incident's title/description; triage builds its payload from Incident fields, IOCs, and assets).
- **Synthetic "absence" Alert/Finding** — fabricate one Alert ("expected X, saw 0") so the rest of the pipeline has a row to chew on and "every Incident has ≥1 Alert" stays true. Rejected: it directly contradicts the glossary, where a **Finding** is "a single raw Wazuh document that matched" and an **Alert** is a real source signal. An Absence Firing corresponds to *no* document; inventing one is a lie in the data model.
- **Detection Suggestion instead of an Incident** — surface absence in the inbox for an analyst to accept. Rejected: an Absence Firing is a *confirmed* condition (the documents are provably not there), not the fuzzy LLM grouping that **Detection Suggestion** denotes.

## Consequences

- **Incidents may have zero linked Alerts.** This was previously always-true-but-incidental, not an enforced invariant. Code that assumed ≥1 Alert (e.g. triage payload construction) must degrade gracefully when an Incident has none.
- **Dedup is reused unchanged.** Absence reuses the one-live-incident-per-`(rule, key_value)` rule: while an open Absence Incident exists for `(rule, none)`, subsequent runs that re-observe the absence are a no-op (the persisting silence folds into the open Incident); a new Absence Incident is only created after the prior one closes. Identical mental model to a positive firing's live-firing dedup, but with an absence branch that creates/short-circuits instead of absorbing Alerts.
- **`correlation_key = none` only (v1).** A terms aggregation can only surface key values that *have* documents, so the engine cannot enumerate *which* keys went silent — there is no key universe to evaluate per-key absence against. Keyed and multi-leg absence (e.g. "for each host that did X, alert if it did *not* do Y") is a recognised future axis but is deferred; a rule that sets a `lte` operator together with a non-`none` correlation key is **rejected at save time**. This also aligns with the engine, where `correlation_key = none` already always takes the single-leg evaluation path.
- **The single-leg path must actually evaluate the threshold.** Today the single-leg/`none` path fires whenever *any* hits exist and never compares against `leg.count`. To support `lte` it must compute `total_count <op> count`; for `gte` with `count = 1` this is identical to today's "hits exist", so existing rules are unaffected.
- **Default `gte` keeps the migration a no-op.** Every existing leg keeps its current `>=` semantics; the new field defaults to `gte`.
