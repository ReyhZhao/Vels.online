# A Scheduled Search Rule leg may carry a Diversity Constraint — a distinct-value threshold evaluated alongside its document count

A **Scheduled Search Rule** could previously only threshold *how many* documents matched per **Correlation Key** (`doc_count ≥ leg.count`). It had no way to express "the same key spanning N *different* values of some other field" — e.g. multiple successful logins for one user originating from two or more distinct countries within the **Window**. We add a **Diversity Constraint** as an optional property of a **Leg** (`distinct_field` + `min_distinct`): the leg is satisfied for a key only when *both* `doc_count ≥ count` **and** the number of distinct values of `distinct_field` in that key's bucket `≥ min_distinct`. We chose set-cardinality (distinct-value count) rather than a geo-distance/time physics check, leg-level rather than rule-level placement, and a `terms` sub-aggregation rather than a `cardinality` one.

## Considered Options

- **Distinct-value diversity, leg-level, `terms` sub-agg (chosen)** — covers the motivating case ("same user, ≥2 countries") exactly, generalises to any aggregatable field (ASN, `source.ip`, device), composes with the existing per-leg `terms` agg + key-intersection at near-zero extra cost, and returns the *actual* distinct values for the incident.
- **True impossible-travel (distance ÷ time-delta exceeds plausible speed)** — rejected for v1: needs geo-coordinates, per-event pairing, and a distance calc that cannot be done in a single aggregation. Recorded as a deliberate future axis, not a no.
- **Rule-level diversity setting** — simpler UI but cannot say "*this* leg must span countries while *that* leg need not," and sits awkwardly beside the per-leg `count`.
- **`cardinality` sub-agg instead of `terms`** — cheaper and exact enough at country scale, but returns only a number; we would have to issue a second query to name the countries in the incident. `terms` (size-capped) hands us values + counts in one shot.

## Consequences

- **A Leg now has two thresholds of different kinds** — a document-count threshold (`count`) and a distinct-value threshold (`min_distinct` over `distinct_field`). Both must hold. A future reader seeing two thresholds on one leg should read them as AND-ed satisfaction criteria, not alternatives.
- **A Diversity Constraint requires `correlation_key != none`.** Without a grouping key the "distinct values across the org" check is true essentially every window for any multi-site tenant — meaningless noise. Enforced as a hard validation error in the builder and stripped-with-warning in the LLM sanitiser.
- **Routing widens.** The evaluator's aggregation path (formerly `key != none AND legs > 1`) now also fires when a single leg carries a Diversity Constraint, so the canonical single-leg impossible-travel rule runs through the per-key agg path rather than the degenerate fetch-all path.
- **Validation invariants:** `min_distinct ≥ 2` (a floor of 1 is satisfied by every non-empty bucket, i.e. not a constraint); `distinct_field` must be aggregatable (`text` resolved via its `.keyword` subfield, same as the compiler already does for `equals`); and `distinct_field` must not resolve to the same Wazuh field as the correlation key (distinct count of the grouping field within its own bucket is always 1 — a dead rule).
- **The incident names the spread.** Because we use a `terms` sub-agg, the firing carries the distinct values + per-value counts; the incident title and description name them (e.g. "…: alice (3 countries: NL, US, RU)") since the spread is the entire reason the incident exists.
- **LLM authoring is in scope from the start** (it is the primary authoring interface): the draft schema and prompt gain `distinct_field`/`min_distinct`, and `GeoLocation.country_name` is curated into the always-present grounding `core_fields` so the headline use case does not depend on field-expansion luck. The mapping-aware sanitiser strips an invalid Diversity Constraint and warns — and because the constraint is load-bearing, that warning is worded to make the semantic loss unmistakable. A warning-*severity* system was considered and deliberately deferred (the flat warning list is unchanged).
- Builds directly on [ADR-0006](0006-scheduled-search-rules-pull-engine.md) (pull engine) and [ADR-0007](0007-search-rule-dynamic-field-catalog.md) (live mapping as catalog); the `distinct_field` is validated against the same live mapping as every other field.
