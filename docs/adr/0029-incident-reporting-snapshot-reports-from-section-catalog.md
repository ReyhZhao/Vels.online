# Incident Reports are immutable snapshots rendered from a SOC-authored section-catalog template, gated by an Audience visibility floor

We want analysts to generate and export a shareable document about an **Incident** — for a customer or an internal stakeholder — the way TheHive's report templates do ([CONTEXT.md](../../CONTEXT.md) → Reporting). The hard part is not rendering a PDF; it is doing so over an Incident that is dense with **internal** material (AI **Triage** tool-traces, internal **Comment**s, raw **Exposure** detail, IOC enrichment internals) without leaking any of it to a customer. This ADR records the architecture that makes a customer-facing **Report** leak-safe *by construction* rather than by author discipline.

## Decision

**A Report is an immutable, point-in-time snapshot, not a live view.** Generating a Report freezes the Incident's then-current state into a stored, audited artifact (the rendered PDF is the canonical "exactly what we shared, and when"). The Incident changing later never alters an existing Report — you generate a new one. A Report is a standalone first-class entity, not a **Comment** or an **IncidentEvent**.

**Templates are composed from a fixed, server-defined Section catalog — never raw markup.** A **Report Template** is an ordered set of **Report Section**s drawn from a server-controlled catalog (Executive Summary, Incident Details, Timeline, IOCs, Actions Taken, Recommendations, Asset Impact) plus free-text intro/outro blocks. The SOC author *selects and orders* sections and fills the free-text blocks; they do **not** hand-write HTML/Django templates that pull arbitrary Incident fields. This is the crux of the leak-safety story: each catalog Section is backed by a server partial that already applies the visibility floor, so a template *cannot* be authored that renders an internal comment body. Raw-template freedom (TheHive's model) is rejected precisely because it reopens that surface.

**Audience is a property of the Template and fixes the visibility floor — leak-safety is a template-author decision made once, not a per-generation toggle.** A `customer` template always renders through the **customer perspective**: the exact floor an **Organization** member would see for that Incident, reusing the existing `incidents/services/visibility.py` rules (non-internal comments/events only; nothing at TLP:RED; no comments/events at TLP:AMBER) — **regardless of which staff member clicks generate**. Because a Report is generated by staff, naively rendering through the generating analyst's own (staff) perspective would leak everything; the `customer` floor is therefore applied unconditionally at render time. An `internal` template renders full fidelity for SOC management and the internal post-incident record.

**Two disclosure rules sit beyond the Audience floor.** The **IOCs** Section respects a **PAP ceiling** — indicators render only at PAP:WHITE/GREEN and are suppressed at PAP:AMBER/RED — so the Permissible Actions Protocol marking governs the one place indicators leave the platform, distinct from the TLP floor that governs everything else. The **Asset Impact** Section renders linked **Asset** names/roles only and **omits all Exposure specifics**: a customer report must not double as an exposure map of the customer's own raw-NAT'd hosts, even though that is the customer's own data and not a cross-tenant leak.

**The Executive Summary is LLM-generated at render time, but grounded only on the audience-filtered content.** A report without a good written summary is worthless, so the Executive Summary Section's prose is generated by the existing LLM infra at generation time. Critically, it does **not** reuse `incidents.llm.grounding.build_incident_grounding` (that function is for the staff Incident Assistant and includes *all* comments/tasks with no floor). The summary grounds on a **visibility-filtered** grounding matching the Template's Audience, so a `customer` summary cannot mention internal findings. The generated prose is frozen into the snapshot like every other Section — the model is never re-run on a later view.

**Generation is a human action or an Incident Assistant proposal — never unattended.** A human analyst generates a Report from the incident detail view at any incident state (an interim update is valid). The **Incident Assistant** may *propose* generating one (joining its propose-and-confirm set under [ADR-0012](0012-incident-assistant-relaxed-action-authority.md), because a customer-facing document is outward-facing), which the analyst confirms. The **Triage Agent** does **not** generate Reports — a customer-facing deliverable stays a human-ratified action even under the agent's unattended autonomy ([ADR-0025](0025-triage-agent-unattended-autonomy-boundary.md)). Auto-generation on close is deferred.

**Persistence is a standalone model with a denormalized template snapshot; rendering reuses the Attachment storage backend.** The `Report` model stores the rendered PDF (via the same object storage as `incidents/services/attachments.py`) plus metadata captured at snapshot time: `incident`, `template` (`SET_NULL` on delete), and **denormalized** `template_name`, `audience`, `tlp`, `incident_state`, `generated_by`, `generated_at`, and a human reference id. Editing a template never touches existing Reports; deleting a template leaves them intact (the denormalized fields preserve the audit record). The PDF is *not* a regular `Attachment` row (those carry their own `is_internal` UX) — a Report is its own entity that merely borrows the storage backend. Format is PDF via WeasyPrint from an HTML intermediate.

**Delivery is store + staff-download + customer-portal surfacing.** Every Report is staff-downloadable. A `customer`-Audience Report additionally surfaces in the Organization member's own incident view for self-service download, gated by the existing `can_view_incident` rule. An `internal` Report is staff-only and never shown to org members. In-platform email delivery to **contacts** is a deferred v2 nicety.

## Considered Options

- **Section-catalog templates with a template-level Audience floor (chosen)** — leak-safety is structural: an author cannot compose a template that renders internal content, and the floor reuses the already-correct `visibility.py`. Cost: less authoring flexibility than free-form templates, and a server code change to add a new section kind.
- **Raw HTML/Django report templates with placeholders (TheHive's model, rejected)** — maximally flexible and familiar, but it makes leak-safety an author-discipline problem: one `{{ comment.body }}` over an internal comment leaks it into every customer report forever. Unacceptable given how much internal AI/exposure data lives on an Incident.
- **Per-Report hand-curation (analyst picks/edits content per report, rejected for v1)** — gives the analyst final say on what ships, but reintroduces a manual leak surface and per-report effort; the template + floor model gets a safe report from one click. A curation step is a possible later layer *on top of* the floor, never instead of it.
- **Live always-current report view (rejected)** — cheaper (no stored artifact), but it makes "what did we tell the customer on June 3" unanswerable, which is the whole point of a shared deliverable. Snapshots are the audit record.
- **Render through the generating analyst's perspective (rejected)** — the obvious naive implementation; it leaks everything because the generator is staff. The `customer` floor must be applied unconditionally.
- **Deterministic Executive Summary = `incident.description` verbatim (rejected for v1)** — simplest and leak-trivial, but a verbatim field dump is not a stakeholder summary; the LLM summary on the audience-filtered grounding was chosen for quality, accepting the model call and the discipline of a separate filtered grounding.
- **Per-org / tenant-authored templates (deferred)** — a tenant authoring the customer-facing layout the SOC sends *them* is backwards and reopens leak-safety; templates are SOC-authored and global (`org = null`) like a System Rule. Per-org variants are a later axis.

## Consequences

- **A new visibility-filtered grounding is required.** The Executive Summary cannot reuse `build_incident_grounding`; a parallel grounding that runs the `visibility.py` filters for the target Audience must be built and kept in sync as the floor evolves. This is the single most important invariant to protect in review: any new Section or grounding field must pass through the Audience floor.
- **The Section catalog is a server-side extension point, not user data.** Adding a report Section is a code change (a new partial that respects the floor), not a template-author action. This is the deliberate trade that buys leak-safety; do not "improve" it into a free-form template engine.
- **Immutability shifts the burden to regeneration.** There is no "edit this report" — corrections are new snapshots. The human reference id and `generated_at` let a stakeholder distinguish versions; superseded reports are retained, not deleted, as the audit trail.
- **TLP changes interact with already-shared reports.** A Report snapshots the `tlp` at generation; later raising an Incident to TLP:RED does not retract a `customer` Report already generated and downloaded. Retraction/expiry of shared reports is out of scope for v1.
- **Staff-only authoring, customer-readable output.** Org members never author or see `internal` Reports; widening either is a later change the Audience field already accommodates.
