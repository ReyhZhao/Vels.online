# The LLM assistants run an agentic tool-calling loop with native web search

The **Incident Assistant** ([ADR-0008](0008-incident-assistant-propose-and-confirm.md)) and the **Rule Draft** assistant ([ADR-0005](0005-rule-author-assistant-ephemeral-stateless.md)) were single-shot: server-precomputed grounding stuffed into the prompt, one model call, a JSON envelope back. To let the model *investigate* — search the internet for threat intelligence and look up related app data (incidents, alerts, assets) on demand — both assistants gain a shared **agentic tool-calling loop**. This is the loop issue #402 identified as the prerequisite for everything; we adopt the loop and web search now, but **defer** converting the search-rule catalog/field handoff to tools (the two-pass flow of ADR-0007 stays).

## Decision

- **One shared orchestrator owns the loop**; providers shrink to a thin `chat(messages, tools) -> (text | tool_calls)` primitive that only translates to/from each SDK. The orchestrator owns the tool registry, server-side tool execution, the iteration cap, and assembly of the result. This keeps the agentic logic written once and the provider abstraction from fragmenting — the core risk flagged in #402.
- **Two-phase per turn.** *Phase 1 — research:* the loop runs with read tools (and web search) until the model stops calling tools or the cap is hit. *Phase 2 — synthesis:* a final tool-free call produces the existing structured envelope (`proposed_actions` for incidents, `draft_rule` for search rules) using the gathered context. This sidesteps the `google_search` ⊥ `response_mime_type=json` incompatibility by construction, keeps the existing parsers/validators (`_parse_assistant_result`, the draft sanitizer) unchanged, and keeps phase-1 within the tool-count budget.
- **Hybrid grounding.** The cheap, always-relevant context is still precomputed server-side every turn (the incident's own state; the search rule's core fields + correlation keys + severities). Tools cover only what is expensive, unbounded, or external.
- **Web search is native per provider.** **Ollama Cloud** (primary runtime) exposes web search as a client-invoked tool (`client.web_search` / `web_fetch`) the orchestrator runs like any other tool. **Gemini 3** (backup) uses native `google_search` grounding combined with our function tools in one request — a **Gemini 3** capability; Gemini 2.5 cannot combine the two, so this ADR requires `GEMINI_MODEL` to be a Gemini 3 model. Self-hosted Ollama (no cloud key) has **no** web search and is a documented degraded mode (a Brave/Tavily fallback is deferred).
- **Egress is PAP-gated.** Web search obeys the incident's **PAP** (Permissible Actions Protocol — what actions may be taken on the information, distinct from TLP's disclosure control). At PAP white/green/amber the model may put incident-specific terms (IOC values, hostnames) into queries. At **PAP:RED** only generic research is allowed: web search stays enabled, but the tool executor **rejects any query containing one of the incident's own indicators** (IOC values, asset/agent names, IPs, linked usernames) by literal-token match, returning a "blocked: PAP:RED" tool result the model works around. The search-rule assistant's web search is generic (threat research, not incident-bound) and is unrestricted.
- **Tool reads reuse existing authorization — never a backdoor.** App-lookup tools call the same permission predicates as the REST endpoints (`filter_incidents_for_user`, etc.). Model-supplied args may *filter* but never *widen* beyond what the caller may see. Default scope is **narrowed to the incident's / draft's organization**, honoring the tenant-isolation invariant; a staff caller may explicitly widen to cross-org (a right they already hold) for cross-tenant threat correlation. All app-lookup tools are strictly **read-only**.
- **Blocking with hard caps (v1).** The endpoint stays a synchronous POST and returns one JSON when the loop finishes, bounded by max iterations (~5), per-tool timeout (~10s), and an overall deadline (~45s, under any proxy limit). The response carries a `tool_trace` for transparency. SSE streaming is deferred to a tracked v2 issue — see [ADR-0014](0014-incident-assistant-sse-streaming.md) for the v2 streaming decisions.
- **Tool count.** The incident assistant carries more than the ~3 tools weak self-hosted models tolerate; we rely on the stronger Cloud models (Ollama Cloud gpt-oss/qwen3, Gemini 3) plus tight single-purpose schemas, and validate reliability in a prototype before committing.

## Considered Options

- **Shared orchestrator + thin provider primitive + two-phase (chosen)** — one loop, identical semantics across providers, existing structured-output contracts preserved, the JSON/grounding conflict dissolved.
- **Loop inside each provider** — lets each provider use its strengths freely, but duplicates the agentic logic and tool execution per provider: exactly the fragmentation #402 warned against.
- **Write-intent tools (actions emitted as tool calls)** — elegant single-phase design, but explodes the tool count past the reliability ceiling on the primary (Ollama) runtime and forces a redesign of the action contract. A clean future evolution once strong models are universal.
- **Uniform custom web_search for all providers (Brave/Tavily)** — one search path, but forgoes the free, native search both primary providers already offer and adds an external dependency/cost. Kept only as the self-hosted fallback.

## Consequences

- `GEMINI_MODEL` must move to a Gemini 3 model; the other LLM features sharing it (triage, correlation, residual grouping, sample-gen) must be re-validated against the new model.
- The provider interface gains a low-level tool-calling primitive; high-level methods (`assist_incident`, `draft_search_rule`) become orchestrator-driven rather than provider-driven.
- The ephemeral/stateless contract holds for the *conversation*, but phase-1 tool execution now has read side effects (queries) and, for the incident assistant, write side effects for the auto-executed action set — see [ADR-0012](0012-incident-assistant-relaxed-action-authority.md).
- Self-hosted Ollama loses web search; deployments needing it must use Ollama Cloud or wait for the deferred external-search fallback.
- Latency rises from one call to a multi-step loop; v1 accepts a longer spinner, v2 adds streaming.
