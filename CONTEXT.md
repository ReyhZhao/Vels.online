# Vels Online — Security Operations Context

The domain language for turning multi-source security signals into actionable
investigations: alerts arrive, get correlated, and become incidents that analysts work.

## Language

### Alerts & detection

**Alert**:
A single security signal from one source (Wazuh event, vulnerability, agent finding, inbound email, API). Has a `source_kind`, free-form `source_ref`, severity, and state.
_Avoid_: Event (reserve for raw Wazuh events upstream), Finding.

**Correlation Rule**:
A user-defined rule that promotes a combination of alerts to an Incident when its legs are all satisfied for the same correlation key within a time window.
_Avoid_: Detection, Pattern, Filter (too vague — a rule is specifically multi-leg + key + window).

**Leg**:
One sub-condition of a Correlation Rule, matching a distinct *class* of alert by its fields (e.g. `title CONTAINS "port scan"`). A rule has one or more legs.
_Avoid_: Clause, Condition (reserve "condition" for the field-level predicate inside a leg).

**Entity**:
A normalised, queryable value carried on an Alert in its **entity envelope**, named per Elastic Common Schema (`host.name`, `source.ip`, `user.name`, `file.hash.sha256`, `process.name`). Sources populate the envelope (required on the ingestion API; canonicalised on ingest); the platform does not extract entities itself. Legs across different `source_kind`s join on shared Entities. See [ADR-0001](docs/adr/0001-ecs-entity-envelope-at-ingestion.md).
_Avoid_: Field, Attribute (those are raw `source_ref` keys, pre-normalisation).

**Correlation Key**:
The Entity type a Correlation Rule binds its legs on. Per-rule choice (`agent`, `source_ip`, `username`, …, or `none` = correlate across all alerts org-wide). `none` rules are typically single-leg.
_Avoid_: Group key, Join key.

**Window**:
The rolling time span within which all of a rule's legs must be satisfied to trigger.

**Firing**:
A single triggering of a Correlation Rule for one Correlation Key value, recorded in a ledger as `(rule, entity_value, incident, fired_at)`. At most one *live* (open-incident) Firing exists per `(rule, entity_value)`; while it is live, further matching alerts link into its incident rather than spawning a new one. A new Firing is only allowed after the prior incident closes.
_Avoid_: Match, Hit, Trigger (use "trigger" only as the verb).

**System Rule**:
A Correlation Rule authored centrally by the Vels SOC (`organization = null`) and applied to every tenant as baseline detection. Tenants cannot edit it but can **mute** it for themselves. See [ADR-0003](docs/adr/0003-system-baseline-plus-per-org-overlay-rules.md).

**Org Rule**:
A Correlation Rule a tenant authors for its own organisation (`organization` set).

**Mute**:
A per-org disablement of a System Rule — the rule stops evaluating for that tenant only, without being deleted globally.
_Avoid_: Disable, Suppress (suppress could be confused with Firing dedup).

### Assets & exposure

**Asset**:
A customer thing the SOC tracks and links to **Incidents** — always owned by exactly one **Organization**. Has a `kind`: a **Host Asset** or a **Route Asset**. Carries common fields (name, `is_active`, `is_permanent`, `last_seen_at`) and is linked to incidents via `IncidentAsset`.
_Avoid_: Entity (reserved for the ECS value on an Alert's envelope), Agent (a Host Asset *wraps* a Wazuh agent but the Asset is the platform's own record, not the agent itself).

**Host Asset**:
An **Asset** representing one of an org's hosts — a Wazuh **agent** by `agent_name` (unique per org), with an optional `ip_address` and `role` (workstation, server, …). The thing that can be **Internet-facing**.

**Route Asset**:
An **Asset** that is a 1:1 wrapper around an `ingress.Route` — the org's *public front* (an FQDN proxied through the protected ingress/WAF). A Route Asset is internet-facing by its very nature; it is the front *through* which a **Host Asset** may be exposed, not the host itself.
_Avoid_: treating the Route Asset as the internet-facing host — the host behind it is a separate **Host Asset**.

**Internet-facing** (a derived status of a **Host Asset**):
Whether a host is reachable from the public internet. It is **not a free-standing boolean an operator toggles** — it is *true exactly when the host has one or more **Exposures***. Surfaced to the LLM-assisted flows (the **Hunt** in-scope asset inventory, the **Incident Assistant**) so they can reason about which customer hosts present external attack surface.
_Avoid_: a flat per-Asset boolean (does not capture *how* a host is exposed, nor multiple simultaneous routes); applying it to a **Route Asset** (a route is the front, not the exposed host).

**Exposure** (a way a **Host Asset** is internet-facing):
One distinct path by which a **Host Asset** is reachable from the internet — a unifying *read-model* over two differently-backed kinds, not a single uniform table. **Ingress-route exposures are derived-on-read** from `Route.backend_asset` (the route FK is their only source of truth — nothing extra is stored); **direct-NAT exposures are persisted rows** (the app has no other record of them). A host has zero or more Exposures; zero ⇒ not **Internet-facing**. Each Exposure is one of two kinds: an **ingress-route exposure** (derived from an *explicit* link between an `ingress.Route` and this **Host Asset** — the app best-effort *suggests* the link by matching the route's `backend_host`, but the recorded link is authoritative, auto-applied only on an *unambiguous* match and kept in sync as either the route *or* the host asset changes) or a **direct-NAT exposure** (operator-declared, the app has no knowledge of it; **one per forwarded service** — a host with RDP and HTTPS NAT'd holds two — each naming the protocol and port that is open). "Servers can be internet-facing via multiple routes" ⇒ a host may hold several Exposures at once.

Every Exposure carries a first-class **protection** trait the LLM-assisted flows reason over: an ingress-route exposure is **protected** (the service sits behind the BunkerWeb reverse proxy / WAF), a direct-NAT exposure is **raw** (the service hangs directly on the internet, no WAF). "RDP NAT'd raw" is a materially more alarming finding than "HTTPS behind the WAF," so the protection trait — not just the fact of exposure — is what the **Hunt** inventory and **Incident Assistant** surface, alongside the per-exposure specifics (route fqdn + backend port, or NAT protocol/port + optional public IP). See [ADR-0023](docs/adr/0023-internet-facing-host-exposure-model.md).
_Avoid_: conflating an Exposure with the **Route Asset** (the Route Asset is the public FQDN record; an ingress-route Exposure is the *link* asserting a given Host Asset sits behind that route).

### Scheduled search

**Scheduled Search Rule**:
A user-defined rule that, on a schedule, pushes its pattern-match down into the Wazuh **OpenSearch** backend to detect a pattern of raw Wazuh alerts *without* ingesting all of them into the platform. Authored with the same manual + LLM-assisted builder as a **Correlation Rule**, but it is a distinct concept: it evaluates by *pull* (a periodic query) rather than by *push* (reacting to an ingested **Alert**), it matches over the raw Wazuh document schema rather than the normalised **entity envelope**, and it has no in-app **Alert** rows until it triggers. Each **Leg** carries a **Count Operator** (`gte`, the default, or `lte`) governing how its document-count threshold is compared — `gte` is the ordinary "at least N matched" detection; `lte` is the inverse, an **Absence Firing**. See [ADR-0006](docs/adr/0006-scheduled-search-rules-pull-engine.md) and [ADR-0007](docs/adr/0007-search-rule-dynamic-field-catalog.md).
_Avoid_: Correlation Rule (reserved for the in-app, push, multi-leg engine over ingested Alerts), Saved Search (a Scheduled Search Rule is evaluated and can raise findings, not just stored). Note: **Hunt** is now a distinct defined concept (the interactive LLM threat-hunting module) — a Scheduled Search Rule is its opposite (non-interactive, scheduled, *pull*), so keep the two apart.

**Finding**:
A single raw Wazuh document that a **Scheduled Search Rule**'s query *or a **Hunt**'s lens* matched — either an **alert** document (`wazuh-alerts-*`) or an **IT Hygiene Inventory** state document (`wazuh-states-inventory-*`, e.g. a specific installed package on a specific host the model judged compromised). When a Scheduled Search Rule triggers, each Finding is **materialised** as an in-app **Alert** (the matches are ingested on trigger — only the matches, never the whole index), and the run's Findings together produce one **Incident**. A **Hunt** reuses the identical substrate and materialise bridge: its matched docs are Findings, grouped by affected org, and each group becomes a **propose-and-confirm** Incident the staff member confirms (materialised on confirm). A behavioral Hunt's *aggregate* insight (e.g. "host X: 400 failed logins then a success") is the model's narrative *over* a set of Findings — the underlying docs remain the materialisable evidence; it is not a separate concept. Almost always there is no pre-existing app Alert for a Finding, since the platform does not ingest the full Wazuh stream. An **Absence Firing** (below) is the inverse case: it triggers precisely *because* no document matched, so it produces **no** Findings at all.
_Avoid_: Hit, Match (use "match" only as the verb for the query selecting a document).

A Scheduled Search Rule is a **System Rule** (org = null, baseline applied to every tenant, per-org **Mute**) or an **Org Rule** (org set) — the same tiers as a Correlation Rule. **Tenant-isolation invariant:** every query is scoped to one organisation's Wazuh agents (`agent.id` ∈ the org's `wazuh_group` members), and a rule's **Window** / **Correlation Key** join *never* crosses tenants — even `correlation_key = none` correlates only within a single org's agent scope. A System Search Rule achieves this by fanning out per org (excluding mutes) at run time.

**Diversity Constraint**:
An additional satisfaction requirement on a **Leg** of a **Scheduled Search Rule**: the leg's matching documents, grouped by the rule's **Correlation Key**, must span at least M *distinct values* of a chosen field (e.g. ≥ 2 distinct `country` for the same `user.name`). It is a property of a Leg, evaluated *alongside* the leg's document-count threshold — both must hold for the leg to be satisfied for a key. It expresses "spread across N different X," which the plain count threshold (which only counts *how many* documents matched) cannot. The motivating case is multiple successful logins for one user originating from two or more different countries within the **Window**. See [ADR-0009](docs/adr/0009-leg-level-diversity-constraint.md).
_Avoid_: Condition (reserved for the field-level predicate *filter* inside a leg — a Diversity Constraint is an aggregation threshold, not a filter), Impossible Travel (that connotes a geo-distance ÷ time physics check, a deliberately deferred future axis — a Diversity Constraint is pure set-cardinality with no geography or distance).

**Novelty Constraint** (informally, _first-seen_ / _new-terms_):
An additional satisfaction requirement on a **Leg** of a **Scheduled Search Rule** that fires on a **value the engine has not seen before**: grouped by the rule's **Correlation Key**, the leg is satisfied for a key only when a matching document carries a value of the leg's chosen **novelty field** whose *earliest* occurrence within a **baseline lookback** lands inside the **Window**. The motivating case is a user logging onto a host that is new *for them* — **Correlation Key** = `user.name`, novelty field = `host.name`. It is the **baseline-comparing** sibling of the **Diversity Constraint**: where a Diversity Constraint counts *distinct values within* the **Window** (pure set-cardinality, no history), a Novelty Constraint compares the **Window** against the **baseline lookback** to find values that are *new to history* — the engine's first axis that reaches outside a single rolling window. Evaluated statelessly as a pushed-down OpenSearch aggregation (terms-of-terms with a `min(@timestamp)` sub-aggregation over the baseline), so it works on a rule's first run with no warm-up and adds no per-entity write-model state. See [ADR-0021](docs/adr/0021-leg-level-novelty-constraint.md).
_Avoid_: Diversity Constraint (that is cardinality *within* the **Window**; a Novelty Constraint is a *window-vs-baseline* comparison), Anomaly / Baseline Deviation (those connote statistical/ML scoring — a Novelty Constraint is exact set-membership: seen before or not), Absence Firing (the inverse-of-presence case, which is still single-window).

**Materialise**:
To create an in-app **Alert** from a **Finding** at trigger time. The bridge that lets a Scheduled Search Rule reuse the existing Incident / IOC / triage pipeline, which all assume real Alert rows. In v1 a materialised search-alert is **born-linked + suppressed**: created with `source_kind = scheduled_search`, already linked to its rule's Incident. Suppression is two-part — it does not *trigger* streaming evaluation, **and** it is excluded from other rules' streaming window scans — so it participates only in its own Search Rule incident and never seeds streaming cross-source correlation. Its Incident also carries `source_kind = scheduled_search`, keeping it outside the streaming `Supersede` logic. (v2 may relax this — see issue #399.) An **Absence Firing** has nothing to Materialise — there are no matched documents — so it produces an Incident with **no** linked Alerts.

**Absence Firing** (informally, a _dead-man's-switch_):
A **Firing** of a **Scheduled Search Rule** triggered by the *absence* of matching documents within the **Window** — a **Leg** whose **Count Operator** is `lte` (e.g. `≤ 0` = "no firewall logs received in the past hour", an alert that the firewall logging went silent). Unlike an ordinary firing it produces **no Findings and no Alerts**: there are no documents to **Materialise**, so the shortfall *itself* is the evidence, carried in the Incident's description and a `SearchFiring` ledger row. It rides the normal Incident / triage pipeline and reuses the one-live-incident-per-`(rule, key)` dedup, so a persisting absence folds into the open Incident rather than spawning one per run. v1 supports it **only for `correlation_key = none`** — a terms aggregation cannot enumerate *which* keys went silent, so there is no key universe to evaluate per-key absence against; a rule that sets `lte` with a correlation key is rejected at save time.
_Avoid_: Finding (an Absence Firing has none), Materialise (there is nothing to materialise), Detection Suggestion (an Absence Firing produces an Incident directly, not an inbox suggestion for an analyst to accept).

### Detection testing

**Rule Test**:
A named, saved test attached to one **Scheduled Search Rule** — the detection-as-code "unit test" for that rule. It bundles a set of **Sample Documents** and one **Expectation**, and is run on demand to check whether the rule still behaves as its author intended. A rule has zero or more Rule Tests; running one produces a **Test Result**. It tests the *whole rule's* external behaviour (does it fire?), never a **Leg** or **Condition** in isolation.
_Avoid_: "logging rule" (a Rule Test is not a kind of detection rule), Fixture (too generic), Correlation Rule / Scheduled Search Rule (a Rule Test is *about* a rule, it is not itself a rule that detects).

**Sample Document**:
One synthetic raw Wazuh document supplied to a **Rule Test** as input the rule under test is evaluated against. Shaped like a real Wazuh OpenSearch document so the same compiled query/match logic applies. Deliberately distinct from a **Finding** (a *real* matched document from production data) and from an **Alert** (a Sample Document is never materialised or ingested).
_Avoid_: Finding, Alert, Sample Log (prefer "Document" to match the Wazuh document schema the rule matches over).

**Expectation**:
What a **Rule Test** asserts the rule should do with its **Sample Documents** — at minimum *should-fire* vs *should-not-fire* (true-positive / true-negative). Expressed at whole-rule granularity.
_Avoid_: Assertion at leg/condition level (out of scope; the Expectation is about the rule firing, not internal leg satisfaction).

**Test Result**:
The outcome of running a **Rule Test**: pass/fail against its **Expectation**, plus diagnostics explaining *why* (e.g. which **Leg** fell short, which key satisfied/missed, diversity shortfall). The diagnostics are explanatory only — they are not authored **Expectation**s.
_Avoid_: Finding, Firing (a Test Result records a test run, not a production detection event).

### LLM-assisted detection

**Residual**:
Alerts that no static rule promoted or linked — still `new`/unlinked, aged past a settle delay, within a lookback window. The input the LLM safety-net reasons over.

**Detection Suggestion**:
A proposed grouping of Residual alerts the LLM flags as suspicious, carrying a rationale and confidence, surfaced in the inbox for an analyst to accept (→ Incident) or dismiss. In v1 the LLM never auto-creates incidents. See [ADR-0004](docs/adr/0004-llm-residual-safety-net-suggestion-only.md).
_Avoid_: Alert (a Suggestion is about a *group* of alerts and is analyst-facing, not a source signal).

**Rule Draft**:
An unsaved, proposed **Correlation Rule** the LLM assembles from a natural-language detection description (e.g. "alert when a new user is created then logs in within 24h"). Presented pre-filled in the rule builder for a human to review, edit, and save; it is never persisted until saved as a real Correlation Rule, and the LLM never activates a rule itself. Produced by a stateless, multi-turn drafting conversation grounded in the scope's recent alerts; the drafter may also **search the internet** for threat intelligence to inform what to detect (the rule-catalog/field retrieval itself stays the existing two-pass flow — converting it to tools is deferred, issue #402). See [ADR-0005](docs/adr/0005-rule-author-assistant-ephemeral-stateless.md) and [ADR-0011](docs/adr/0011-assistant-agentic-tool-calling-loop.md).
_Avoid_: Detection Suggestion (that is the LLM acting on a *group of past alerts* to propose an Incident; a Rule Draft is the LLM acting on *a human's words* to propose a future-matching Rule). Avoid Suggestion (overloaded with Detection Suggestion).

**Incident Assistant**:
A staff-only, conversational LLM interface embedded in the incident detail view. Answers questions about a specific incident using live server-side grounding (the incident's own fields, linked alerts, IOCs, tasks, available task templates) and can **reach beyond that seed via tools** — looking up related incidents, alerts, and assets in the app, and searching the internet for threat intelligence — to inform its answer. The conversation is ephemeral (client state, not persisted) and the seed grounding is recomputed server-side every turn. Actions split by risk: the assistant **auto-executes a bounded set of internal, reversible, non-lifecycle actions** (e.g. add an internal comment, self-assign, tag, link a known asset, **record findings on a manual task**), each audited; everything externally-visible or lifecycle/severity/disclosure-affecting (state transition, severity/TLP/PAP/subject/assignee change, task-template application, contact message, exception, close) remains a **proposal the analyst confirms**. It can also **work the incident's manual tasks** — researching each, then recording its findings as a task-scoped internal comment — but it **never executes `automated` or `wazuh_response` tasks** (those touch live infrastructure) and **never closes a task itself**: completing a task stays the SOC member's choice. See [ADR-0011](docs/adr/0011-assistant-agentic-tool-calling-loop.md), [ADR-0012](docs/adr/0012-incident-assistant-relaxed-action-authority.md) (which supersedes [ADR-0008](docs/adr/0008-incident-assistant-propose-and-confirm.md)), and [ADR-0013](docs/adr/0013-incident-assistant-works-manual-tasks.md).
_Avoid_: conflating with Detection Suggestion (which is about grouping alerts pre-incident) or Rule Draft (which is about authoring Correlation Rules). _Avoid_: assuming the assistant can autonomously take *any* action — autonomy is bounded to the internal/reversible/non-lifecycle set; higher-risk actions still require human confirmation.

**Hunt**:
A staff-initiated, LLM-assisted threat-hunting investigation — a *new module* distinct from the **Incident Assistant**. Where the Incident Assistant is incident-*bound* (it lives inside one incident and is seeded by it), a Hunt is incident-*producing*: it starts from no incident, seeded by a staff member's free-text question or a link to an external malware/threat report, then ranges over Wazuh telemetry (via the existing org-scoped `OpenSearchClient`/`WazuhClient`, **not** an external MCP server in v1) to find whether the IOCs/patterns it identifies appear in the customer fleet. It reuses the Incident Assistant's plumbing — the shared agentic orchestrator, providers, native web search, and SSE/async streaming ([ADR-0011](docs/adr/0011-assistant-agentic-tool-calling-loop.md), [ADR-0014](docs/adr/0014-incident-assistant-sse-streaming.md)) — but is its own concept with its own grounding, surface, and output authority. See [ADR-0015](docs/adr/0015-threat-hunting-incident-producing-module.md) (the module: incident-producing, persisted, propose-and-confirm, own-clients-not-MCP) and [ADR-0016](docs/adr/0016-hunt-runs-as-background-job-streamed-over-sse.md) (each turn runs as a Celery background job streamed over reconnectable SSE). Unlike the Incident Assistant's *ephemeral* conversation, a Hunt is a **first-class persisted, resumable, auditable entity** (seed, owner, org scope, transcript/tool-trace, findings, spawned-incident links): a staff member can continue a hunt later, review earlier hunts, and the record is the audit trail for cross-tenant access. Its durable transcript *is* its grounding (there is no incident seed to recompute). Every turn (both phases) is additionally briefed with an **in-scope asset inventory** — a per-org, tenant-isolated block of the customer's *own* assets (each org's Wazuh agents by name/IP/OS, plus its ingress-route FQDNs/backends) so the model treats them as known-good and does not flag the customer's own infrastructure as the attacker; it is derived from the resolved scope and `ingress.Route`, not a new asset store. "Resumable" means a human adds more turns — **not** an automatic scheduled re-run (that is a **Scheduled Search Rule**'s job). Every Hunt opens in a **Scoping** phase (below): the model and the staff member refine the seed into a shared understanding — using the full toolset for orientation but committing no Findings — until the human fires an explicit "Begin hunt" gate that starts the authoritative, evidence-committing search. When a Hunt finds something, it does not auto-create incidents: it surfaces findings grouped by affected org and offers a **propose-and-confirm** Incident per org ([ADR-0004](docs/adr/0004-llm-residual-safety-net-suggestion-only.md)'s human-in-the-loop principle holds), **materialising** the matched raw Wazuh docs as Alerts on confirm (reusing the Scheduled Search Rule bridge). **v1 is staff-only and cross-org** (the SOC hunts threats across all tenants, a sanctioned staff right); a later v2 may expose constrained single-org hunting to org members. When a Hunt finds something needing human investigation it does **not** silently spawn an incident — see _Flagged ambiguities_ for the open output-authority question (Detection Suggestion vs proposed Incident vs auto-create).
_Avoid_: Scheduled Search Rule (non-interactive, scheduled *pull* detection — a Hunt is interactive and human-initiated), Detection Suggestion (that is the LLM grouping *existing* unlinked alerts; a Hunt actively *seeks* threats from a seed and reaches into raw Wazuh data), Incident Assistant (incident-bound, not incident-producing).

**Scoping** (a Hunt phase):
The pre-search phase of a question-seeded **Hunt**: an interactive dialogue between the staff member and the LLM that sharpens the seed question into a shared understanding *before* the authoritative search runs. The phase boundary is **whether evidence is committed**, not which tools are available: during Scoping the model has the **full toolset** (web search *and* every Wazuh lens) so it can pull in threat-intel and asset/telemetry context to ask grounded questions, but its lens calls run with a **non-persisting findings sink** — they report counts back to the model only and commit **no Finding** (and therefore propose no Incident). The phase is two-way (the model grills the human and the human corrects the model) and is left only by an explicit human "Begin hunt" gate, which transitions the Hunt into its **Searching** phase — the run that actually commits Findings. Scoping turns are part of the Hunt's durable, auditable transcript.
_Avoid_: **follow-up turn** (the post-completion "dig deeper" continuation — that happens *after* results exist; Scoping is strictly *pre-search*), "grilling session" (informal; the canonical name is Scoping), refinement (overloaded with the follow-up dig-deeper turns).

**IT Hygiene Inventory**:
The current *state* of a Wazuh agent's host — its **installed software**, **running processes**, and **services** (Wazuh also collects OS/ports/hardware/hotfixes) — collected by the agent and stored in Wazuh's `wazuh-states-inventory-*` OpenSearch indices, keyed per `agent.id`. It describes what a host *is and has right now*, not a security event that fired — so it is read through the **OpenSearch** backend (not the syscollector REST API) under the same per-org `agent.id` scoping as every other Wazuh query. Both LLM surfaces read it: the **Incident Assistant** lists one host's inventory (org-scoped to the incident's organisation), and a **Hunt** value-searches it across the fleet ("which hosts run X?"). In a Hunt, an inventory row the model judges malicious becomes a **Finding** only through a **deliberate, explicit recording step** — distinct from the IOC sweep's match-is-evidence auto-recording — because a bare software/service/process *name* match is not itself proof of compromise (most matches are benign). See [ADR-0022](docs/adr/0022-it-hygiene-inventory-tools-state-indices-explicit-recording.md).
_Avoid_: **in-scope asset inventory** (the Hunt's known-good briefing — a *roster* of the org's own agents by name/IP/OS; IT Hygiene Inventory is the *contents* of a host, not the list of hosts), syscollector API (the live REST endpoint — this platform reads the inventory *indices* instead).

**Shared Infrastructure event** (a.k.a. _agentless event_):
A Wazuh document **not linked to any tenant's agent** — it is logged against the Wazuh **manager** (`agent.id = "000"`) rather than an agent in some org's `wazuh_group`. In this deployment these are the perimeter sources shared across *all* tenants: a single **firewall** and a single **reverse proxy** that forward via the manager (syslog), so the document carries **no per-org attribution** at all. This is the deliberate exception to the **Tenant-isolation invariant** (which attributes an event to an org *only* by `agent.id` ∈ that org's `wazuh_group`): a Shared Infrastructure event belongs to no one tenant and to all of them at once. The Scheduled-Search world already names this *agentless/infrastructure mode* (`include_agentless` ⇒ the `agent.id` filter is dropped); threat hunting needs the same reach because perimeter telemetry is essential evidence.

A Hunt attributes such events to the **Infrastructure organisation** (below) and selects them with the positive filter `agent.id = "000"` (never by *dropping* the agent filter — that would glob every tenant's data into one un-attributed query). See [ADR-0017](docs/adr/0017-shared-infrastructure-pseudo-org-for-agentless-hunting.md).
_Avoid_: "an org's firewall" (the firewall/proxy are shared, owned by no tenant), Agent (a Shared Infrastructure event is precisely the *absence* of a per-host agent).

**Infrastructure organisation**:
A dedicated, **non-tenant** `Organization` row (`is_infrastructure = True`) that is the home for **Shared Infrastructure events**. It owns no customer, has no `wazuh_group` members, and is **excluded from every "real tenant" code path** — System Search Rule fan-out, fleet vulnerability scans, per-org incident tasks, billing/dashboards/tenant counts — via a tenants-only queryset. It is visited **only** by **Hunt** scope resolution: an *all-orgs* Hunt includes it automatically, and it is also selectable on its own in the staff hunt-scope picker. Its query scope is special-cased to `agent_ids = ["000"]` rather than resolved through `get_agents(wazuh_group)`. Because it is a real `Organization`, infra **Findings** ride the normal **propose-and-confirm** path and **materialise** into an **Incident** *in the Infrastructure organisation* — keeping a real, agent-bound tenant's incidents free of mis-attributed perimeter events. See [ADR-0017](docs/adr/0017-shared-infrastructure-pseudo-org-for-agentless-hunting.md).
_Avoid_: Tenant / customer org (it represents shared perimeter infrastructure, not a paying tenant — never invite users to it, bill it, or fan System Rules out to it).

### Triage

**Subject**:
The classification of *what an Incident is about* — a named, slugged category (e.g. "Brute Force", "Malware", "Phishing") that an **Incident** carries via a nullable FK. It is the axis the **Triage** Classify phase matches an Incident onto, and it binds the Incident to its **playbook** (the subject's **task template(s)**) — so a matched Subject is what makes the **Triage Agent**'s **Work** phase runnable. Open-ended and operator-managed (Subjects can be archived); an Incident with no Subject yet is **Unclassified** (untriaged, or triaged without a confident match). It is the SOC-analyst sense of an incident's "type".
_Avoid_: **type** (fuzzy and overloaded — when someone says an incident's "type" they almost always mean its Subject, *not* its `source_kind`/origin or `severity`); `source_kind` (that is *where the Incident came from* — Correlation Rule, Scheduled Search, Threat Hunt, Manual — an orthogonal origin axis, not a classification); category (vaguer synonym).

**Triage**:
The **unattended** LLM pipeline that runs automatically on a newly-created **Incident** — it has no human in the loop (it fires seconds after the Incident exists, before any analyst opens it). It runs in two phases. The **Classify** phase is a single, cheap model call that assesses the Incident and emits a disposition: a severity recommendation, a recommended action, a **subject** classification, a **false-positive confidence**, and a **disposition confidence**. High false-positive confidence (≥ the org's threshold) auto-closes the Incident as a false positive; otherwise the **Work** phase may run. The Classify phase always runs; the Work phase is gated. See [ADR-0024](docs/adr/0024-two-stage-triage-classify-then-agentic-work.md).
_Avoid_: **Incident Assistant** (that is *human-in-the-loop*, conversational, and incident-*bound* to one already-open Incident; Triage is unattended and runs once at creation), **Hunt** (incident-*producing* and human-initiated). Triage is distinct from **Detection Suggestion**/**Correlation** (those decide *whether* an Incident exists; Triage acts *on* one that already does).

**Disposition Confidence**:
The Classify phase's confidence that the Incident is *real and correctly classified* (genuine, right subject, right severity) — the positive signal that the **Work** phase gates on, distinct from and not the inverse of **false-positive confidence** (which only measures "this is junk → close it"). The **Triage Agent** runs only when disposition confidence clears the org's **work threshold** *and* a subject was matched (so a **playbook** exists to run).
_Avoid_: false-positive confidence (orthogonal — an Incident can be low-FP yet still low-disposition-confidence, e.g. clearly not junk but ambiguous which subject/severity), severity (a recommendation, not a confidence).

**Triage Agent** (the Triage **Work** phase):
The agentic, tool-using phase of **Triage** that runs *only* on high **disposition confidence** with a matched subject. It reuses the shared agentic orchestrator (the same loop behind the **Incident Assistant** and **Hunt**) to research the Incident and then *act* on it autonomously — but, unlike the Incident Assistant, **with no human present to confirm anything**. Its autonomy is therefore confidence-gated rather than human-gated: it *selects and applies* the matching **playbook**(s) (the matched subject's task template(s)) via its own `apply_task_template` tool call — which for the Triage Agent is an executed write, not the analyst-confirmed *proposal* it is for the Incident Assistant — works the **manual** tasks (researching each and recording findings, never closing them — ADR-0013 holds), sets severity/subject/state, and runs **automated** (Semaphore) tasks. It runs a **wazuh_response** task **only** if that response is explicitly **autonomous-response approved** (below). Its unattended autonomy *also* extends to two outward/lifecycle actions the Incident Assistant only ever *proposes*: it may **notify the incident's contacts** (an LLM-written `send_contact_message`, the same auto-messaging the platform already does on close) and **escalate** (bump severity / page) when research warrants — escalation is permitted precisely because it errs toward *more* human attention. It does **not** create detection exceptions and does **not** close the Incident (closing a real, worked Incident stays the human's ratification; only the **Classify** phase auto-closes, and only false positives). On finishing, it routes to the on-call analyst and lands the Incident in one of two states: **`in_progress`** when work remains (the analyst continues it), or a new **`pending_closure`** state when the agent judges the threat *contained* (automated/`wazuh_response` tasks ran, research recorded) and only human ratification of the open manual tasks remains. Either way the analyst's role becomes *ratifying* the agent's work — reviewing findings, running any human-gated task, and closing — rather than starting from scratch. The run is recorded through existing audit (per-action autonomous timeline events, an AI-triage summary comment carrying the tool-trace and a "what remains" note); it is **not** a live-streamed, persisted entity like a Hunt, because nobody is watching at triage time. See [ADR-0024](docs/adr/0024-two-stage-triage-classify-then-agentic-work.md) (the two-stage execution model) and [ADR-0025](docs/adr/0025-triage-agent-unattended-autonomy-boundary.md) (the unattended autonomy boundary, which relaxes [ADR-0013](docs/adr/0013-incident-assistant-works-manual-tasks.md) for this actor only).
_Avoid_: Incident Assistant (human-in-the-loop, proposes high-risk actions; the Triage Agent *executes* them unattended under a confidence gate), "auto-remediation" (overstates it — destructive infra actions stay behind explicit pre-authorization), proposing (the Triage Agent does not *propose* — with no human present a proposal would just languish; it either acts within its authority or leaves the work for the routed human).

**Autonomous-Response Approval**:
A deliberate, default-off marking on a **wazuh_response** catalog entry that pre-authorizes the **Triage Agent** to run that response *unattended* on high disposition confidence. It is **distinct from** the catalog's existing `requires_confirmation` flag (which only governs whether a *human* clicking "Run" sees a confirm dialog — a UI nicety, never consent for a robot to act). It is the single gate that lets the highest-consequence actions in the system (isolate a host, block an IP) fire with no human. The marking is **global per response** (not per-org, not per-playbook-step): an approved response may auto-fire on *any* tenant's estate, so a response is approved only when it is safe fleet-wide. Trusted **automated** (Semaphore) tasks need no such marking — they auto-run on high confidence — because the SOC authors them and treats the whole automation catalog as safe to fire unattended. See [ADR-0025](docs/adr/0025-triage-agent-unattended-autonomy-boundary.md).
_Avoid_: `requires_confirmation` (the human-facing confirm-dialog flag — overloading it would silently authorize unattended destructive actions nobody signed off on), per-org opt-in (rejected for v1 — the approval is global per response).

**Triage Lesson**:
A distilled, reusable disposition heuristic the **Triage** pipeline learns from **resolved Incidents** and applies to new ones — "for Incidents shaped like this, the SOC has learned X." It is the platform's *self-learning* memory: guidance keyed on a **Subject** (and, secondarily, `source_kind` — the same detection shape from a Correlation Rule vs a Scheduled Search may warrant different handling), not a rule that fires — it **informs** the model's assessment but never itself authorizes an action (every action still passes the [ADR-0025](docs/adr/0025-triage-agent-unattended-autonomy-boundary.md) authority gates, so a wrong Lesson can change only what the model *believes*, never what it *does*). Two tiers mirror the **System Rule** / **Org Rule** split: an **Org Lesson** (`organization` set) is learned and applied within one tenant; a **Global Lesson** (`organization = null`) is SOC-curated, carries **no tenant specifics**, and applies fleet-wide so a small tenant inherits the SOC's cross-fleet experience. A Lesson is only ever distilled from a **human-ratified** closure (never the **Triage Agent**'s own unratified disposition) and is inert until a human approves it (**proposed → active**), keeping the machine from training on its own guesses.
_Avoid_: **Correlation Rule** / **Scheduled Search Rule** (those *detect* and *fire* on live signals; a Lesson only *informs* triage judgement), **memory** (too generic — name the concept), treating a Lesson as an automation trigger (it is a prior, not an authority).

**Precedent**:
A single **resolved Incident** retrieved into a new Incident's **Triage** context as a worked example — "this looks like INC-1234, which was closed as a false positive because X." Precedents feed the cheap **Classify** phase, matched by shared **Entities**/**IOCs** (*not* by **Subject**, so they apply before a subject is chosen), and carry the *final, human-ratified* subject/severity/disposition — flagged when a human overturned the agent's original call — so Classify learns from corrected outcomes and dampens overconfidence. **Strictly per-tenant:** a Precedent is only ever drawn from the *same* **Organization**'s history; raw case data never crosses tenants (cross-org learning flows only through generalized **Global Lessons** — see [ADR-0031](docs/adr/0031-cross-org-triage-learning-isolation.md)).
_Avoid_: **Triage Lesson** (a Precedent is one real past **Incident**; a Lesson is the *generalization* distilled from several), **Related incident** (the existing correlation-search notion of a *concurrently-active* campaign — a Precedent is specifically a *resolved* case retrieved for its outcome).

**Classification Correction**:
A human overturning the **Triage** Classify phase's output on an **Incident** — changing its **Subject**, overriding the severity, or reversing the false-positive/disposition call. It is the platform's strongest *self-learning* signal (a *labeled* correction, stronger than a plain closure): it powers a **Classify-accuracy metric**, enriches **Precedent**s with the corrected outcome, feeds the distillation sweep, and counts as a *contradiction* against any **Triage Lesson**/**Precedent** that drove the wrong call.
_Avoid_: closure (a Correction is about the *classification* being wrong, orthogonal to whether the Incident is closed), **Supersede** (that relinks alerts between Incidents; a Correction re-labels one Incident's classification).

### Live attack map

**Attack** (a map event):
A single inbound security event **projected** onto the **Live Attack Map** — one external party striking the protected estate, drawn as an arc from its **source country** to the targeted organisation. An Attack is a *read-model projection over raw Wazuh events*, **not** a new stored signal: it is any geo-locatable event at or above a **configurable severity floor** (a `rule.level` threshold, deliberately set low while the map is young to maximise density, raised toward "real attacks" later) whose **source** resolves to a *real foreign country* — taken from Wazuh's GeoIP enrichment (`GeoLocation`, which also carries exact coordinates when present), else a firewall decoder's own country field; the firewall's literal `"Reserved"` value marks an internal endpoint and is **not** a country. Its **destination** is the *targeted organisation's own location* (and the **Infrastructure organisation**'s home point for shared-perimeter events), **never** the destination IP's geography. v1 is **inbound only** — internal-origin (`"Reserved"` source) events are out of scope, a deliberately deferred *egress* axis (internal → external, the opposite arc).
_Avoid_: **Alert** (a persisted, ingested in-app signal; an Attack is an ephemeral visualisation projection that is never **Materialised** or stored), **Finding** / **Firing** / **Incident** (detection and lifecycle records — an Attack neither triggers a rule nor opens an incident), **Shared Infrastructure event** (that is the underlying perimeter *document*; an Attack is the geo-projected arc *derived from* such a document or from a tenant agent's event).

**Live Attack Map**:
A **staff-only, cross-org** SOC dashboard that animates recent **Attacks** in near-real-time (Kaspersky-cybermap style): a severity-coloured arc per Attack over a world map, plus rolling top-source-country and top-attack-type panels and a recent-attack feed. Cross-org by the same sanctioned staff right as a **Hunt**, so it sits **outside the Tenant-isolation invariant** — it is never a per-tenant view, and the **Infrastructure organisation**'s shared-perimeter events are in scope. The delivery architecture (one presence-gated shared snapshot streamed to all viewers over SSE, keeping OpenSearch load independent of audience) is [ADR-0027](docs/adr/0027-live-attack-map-presence-gated-shared-snapshot-over-sse.md).
_Avoid_: a per-org dashboard (the Live Attack Map is deliberately global/staff-only — a tenant cannot see the shared perimeter, which carries no per-org attribution), operational observability (it visualises the SOC *security* domain, not platform health — ADR-0019 keeps those apart).

### Collaboration

**Incident Presence**:
The ephemeral, real-time record of which **actors** are currently active on a given **Incident** — both **staff** members who have it open *and* the incident's **AI agents** (the **Triage Agent** on a re-run, the **Incident Assistant**) while they work it — surfaced in the incident detail view so analysts don't unknowingly collide with each other or miss live AI activity. One concept with a single per-incident **roster**: each actor carries a current **activity** sub-state — `viewing` (the default, known passively from the open stream), `editing` a specific **Comment**, or `working` a specific **Task** — not three separate signals. Human actors announce presence by holding the SSE stream and heartbeating; **AI actors don't hold a connection** — their presence is written into the same registry from the shared agentic orchestrator's step callbacks (the [ADR-0014](docs/adr/0014-incident-assistant-sse-streaming.md) `on_event` boundaries) and dropped when the run ends. An AI actor is a *synthetic* roster member (`actor_kind=ai`, e.g. "🤖 Incident Assistant"), not a `User`; the Assistant is **attributed to its invoking analyst** ("run by Dana"). AI actors never contend for the comment lock because they only ever *create* new comments (a Triage re-run appends a fresh `ai_triage` comment, never mutating an existing body). Presence is *declared, not durable*: viewing is implied by an open connection, while `editing`/`working` are activity transitions the client pushes up (the server cannot infer them from the socket alone). It is **staff-only in v1** (customer org members neither see nor emit presence, though they can access the incident) and is **throwaway live state, never an audit record** — it is held in the ephemeral presence registry, **not** written to the **IncidentEvent** timeline.

Presence is **advisory for tasks but a soft lock for editing an existing Comment** — a deliberate asymmetry: `working` a **Task** only *informs* (two analysts may focus the same task; durable `Task.state`/`assignee` are never touched), whereas `editing` an existing **Comment** is *exclusive*, because `Comment.body` is a clobberable shared field (last-write-wins lost-update risk). The first editor holds the lock; others get a read-only editor attributed to the holder. The lock rides the same presence record and TTL — so a dead tab self-heals — plus an **idle auto-release** (~5 min since the holder's last keystroke, refreshed on typing) so a walked-away-but-open editor cannot deadlock the comment. Composing a *new* Comment is never locked.
_Avoid_: **IncidentEvent** (the durable, audited incident timeline — Presence is deliberately *not* persisted there), **IncidentDelegation** / **assignee** (those are durable *ownership* facts; Presence is "who is looking right now", which is orthogonal to who owns the incident), "online status" (Presence is per-incident, not a global who's-online roster).

### Reporting

**Report Template**:
A reusable, SOC-authored definition for rendering an **Incident** into a shareable document — the equivalent of TheHive's report templates. It is the *recipe* (layout, sections, branding, which incident data to pull), authored once and applied to many Incidents; it is never itself a document about any one Incident. Every Report Template declares an **Audience** that fixes the visibility floor of the **Report**s rendered from it. **SOC/staff-authored and global in v1** (like a **System Rule**, `org = null`) — tenants do *not* author their own report templates (per-org template variants are a deferred axis), keeping the section catalog and **Audience** floor SOC-controlled guarantees. Reports carry **SOC/platform branding** with the target **Organization**'s `name` rendered as text ("Prepared for: …"); per-org logo upload is deferred.
_Avoid_: **Report** (the rendered output for one Incident, below), **task template** (a **playbook** of **Task**s applied to an Incident — a Report Template renders a *document*, it does not create work), **Comment** (a Report is assembled into a standalone artifact, not appended to the incident thread).

**Report**:
An **immutable, persisted, point-in-time snapshot** rendered from a **Report Template** against one **Incident** — the customer/stakeholder-facing deliverable. Generating a Report freezes the incident's then-current state into a stored, audited artifact; the Incident changing later does **not** alter an existing Report (you generate a new one). It is the durable record of "exactly what we shared, and when" — never a live always-current view that re-renders on each open. Content is **fully template-driven** (no per-Report hand-curation in v1): the template picks sections, and the incident's **TLP** + per-**Comment** `is_internal` marking are the hard disclosure floor (see **Audience**). A Report is a stored, staff-downloadable artifact; a **`customer`**-**Audience** Report *also* surfaces in the **Organization** member's own incident view for self-service download (gated by the existing `can_view_incident` rule), whereas an **`internal`** Report is staff-download only and never shown to org members. (In-platform email delivery to **contacts** is a deferred v2 nicety.)

A Report may be generated **at any incident state** (e.g. an interim update mid-investigation) and records the **Incident**'s state at snapshot time. Generation is triggered by a **human analyst**, or **proposed by the Incident Assistant** for the analyst to confirm — it is an outward-facing action, so it joins the Assistant's *propose-and-confirm* set (ADR-0012), never auto-executed. The **Triage Agent** does **not** generate Reports (a customer-facing deliverable stays a human-ratified action even under the agent's unattended authority); auto-generation on close is deferred.
_Avoid_: live view / dashboard (a Report is frozen, not recomputed), **Comment** / **IncidentEvent** (a Report is a standalone document, not a thread entry or timeline row), regenerating-in-place (a Report is never mutated; a fresh snapshot is a new Report).

**Audience** (of a **Report Template**):
Which visibility floor a **Report** renders through — `customer` or `internal` — fixed on the **Report Template**, *not* a per-generation toggle the analyst can fat-finger. A **`customer`** template always renders through the **customer perspective**: the exact floor an **Organization** member would see for that **Incident** (the existing `incidents/services/visibility.py` rules — non-internal **Comment**s/**IncidentEvent**s only, gated by **TLP**: nothing at TLP:RED, no comments/events at TLP:AMBER), regardless of which staff member clicks generate — so the floor cannot be defeated by the generator being staff. An **`internal`** template renders **full fidelity** (all comments, all events, AI tool-traces) for SOC management / the internal post-incident record. Leak-safety is therefore a property of the *template*, chosen once by the SOC author.
_Avoid_: per-Report visibility override (v1 has none — the Audience is the template's), rendering through the generating analyst's own (staff) perspective (that would leak internal content into a `customer` Report — always render a `customer` Report through the org-member floor).

**Report Preview**:
A **live, unsaved, never-persisted, staff-only** rendering of what a **Report** from the selected **Report Template** *would* look like for this **Incident** right now — an authoring aid the analyst uses to compose a good report *before* freezing the immutable snapshot. A Report Preview is explicitly **not** a **Report**: it has no `reference_id`, is never stored, never produces a PDF, never records an **IncidentEvent**, and is **never shown to org members / customers** (only the frozen `customer` Report is). It exists precisely so the **Report** can stay a pure snapshot — the live thing is the Preview, so the artifact never has to be live. The Preview renders the deterministic Sections **read-only** through the **same Audience floor** as generation (`build_report_grounding`); a staff member previewing a `customer` template sees the customer floor, never staff fidelity. The four free-text blocks — `intro_text`, `outro_text`, `recommendations_text`, and the **Executive Summary** prose — are **editable in the Preview** (pre-filled from the template, or from the LLM for the summary), and the analyst's edits **freeze into that one Report's `content` on generate — never written back to the global Template** (a per-Report override, not a template mutation). The Executive Summary is generated **on demand** in the Preview (one LLM call on the audience-filtered grounding), then editable; a previewed summary is reused **verbatim** at generate (no second model call). A `customer` Preview on a **TLP:RED** Incident renders **no body at all** — a refusal placeholder + disabled generate, the same early-out as generation — so a customer body is never rendered for a RED incident even in preview.
_Avoid_: **Report** (the Preview is the *draft*; the Report is the frozen artifact — never call a Preview a Report or vice-versa), live Report view (the **Report** is still never live — the *Preview* is the live surface, a distinct concept), writing edited free-text back to the **Report Template** (edits are a per-Report override only).

**Report Section**:
One building block of a **Report Template**, drawn from a **fixed, server-defined catalog** plus free-text intro/outro blocks. v1 catalog: **Executive Summary** (LLM-generated, below), **Incident Details** (structured fields — title, severity, subject, state, dates, TLP), **Timeline** (audience-filtered **IncidentEvent**s), **IOCs**, **Actions Taken** (completed **Task**s incl. automated/`wazuh_response` outcomes), **Asset Impact**, and **Recommendations** (a template free-text block in v1, *not* LLM-generated — the summary is the one AI surface for now). Two disclosure rules beyond the **Audience** floor: the **IOCs** Section respects a **PAP ceiling** — IOCs render only at PAP:WHITE/GREEN and are suppressed at PAP:AMBER/RED (so the indicator-handling marking *means something* where indicators leave the platform); and the **Asset Impact** Section renders linked **Asset** names/roles only, **omitting all Exposure specifics** (a customer report is not an exposure map of the customer's own raw-NAT'd hosts). A template is *composed* of ordered Sections — the SOC author selects and orders them and fills the free-text blocks; the author does **not** hand-write markup that pulls arbitrary **Incident** fields. Each catalog Section is backed by a server-controlled partial that already applies the **Audience** visibility floor, so leak-safety is enforced by the catalog rather than trusted to the author. This is the deliberate reason the template language is a section catalog and *not* a raw HTML/Django template: a free-form template could render an internal **Comment**'s body and defeat the floor.

The **Executive Summary** Section is special: its prose is **LLM-generated at render time** (a report without a good written summary is worthless), reusing the existing LLM grounding/orchestrator infra — but **grounded *only* on the same visibility-filtered content the Section catalog exposes for the template's Audience**, so a `customer` summary cannot mention internal findings. The generated prose is then **frozen into the Report snapshot** like every other Section (the model is not re-run on later views). See [ADR-0029](docs/adr/0029-incident-reporting-snapshot-reports-from-section-catalog.md).
The free-text blocks (`intro_text`, `outro_text`, `recommendations_text`, and the editable **Executive Summary**) accept a **constrained rich-text subset** — bold, italic, underline, bullet/ordered lists, and capped indent — *not* free-form markup. The subset is enforced **structurally**: stored as a tiny sanitized-HTML allowlist (`p, br, strong, em, u, ul, ol, li` + a fixed `indent-N` class, **zero attributes, no `style`/links/images**) run through a server-side sanitizer (`nh3`) on write, so it cannot express `<script>` or pull any **Incident** field. This is *formatting*, distinct from the rejected raw-template freedom below (which is about rendering arbitrary incident data and defeating the floor).
_Avoid_: raw template / free-form markup (rejected for v1 — it reopens the leak surface the **Audience** floor closes; the constrained rich-text *formatting* subset above is **not** this), **task template** (unrelated — that is a **playbook** of work).

### Partner intake

**Connection**:
A configured, per-external-organisation email channel through which an outside party feeds **Incidents** into the platform over email — the machine-to-machine sibling of the human-facing **Contact** thread. Holds the partner's sending email address(es), a **field mapping** (how email fields map onto Incident fields — **TLP**, **severity**, subject, …), a **matching** config (how an inbound email is tied to an existing Incident — an **External Reference** in the subject), and a **direction** (inbound-only or **bi-directional**). Each Connection names the target **Organization** its Incidents belong to. Two kinds: a **CSIRT/peer Connection** (a peer SOC feeding *one customer org*, usually bi-directional) and a **Vendor Connection** (a supplier's security bulletins, inbound-only, feeding the **Infrastructure organisation**). See [ADR-0032](docs/adr/0032-partner-intake-direct-incidents-over-verified-email.md) (direct-Incident-over-verified-email) and [ADR-0033](docs/adr/0033-partner-sync-mirror-external-comments-tlp-red-gate.md) (bi-directional comment-mirror gating).
_Avoid_: **Contact** (a named human at a customer org the SOC messages *about* an incident — a Connection is a machine-to-machine feed *from* a peer organisation), Integration (too generic), Feed (biases toward the broadcast/bulletin case only).

**Partner** (an Incident `source_kind`):
The origin marking on an **Incident** that entered the platform through a **Connection** — a peer CSIRT detection or a supplier bulletin over email. A Partner Incident is created **directly**: unlike the phishing **inbound_email** flow it never becomes an **Alert** and never passes through correlation, because a peer SOC's detection is already a fully-formed incident, not a raw signal. Distinct from **external** (the generic API-created incident). See [ADR-0032](docs/adr/0032-partner-intake-direct-incidents-over-verified-email.md).
_Avoid_: **inbound_email** (transport-named and reserved for the phishing **Alert** path), **external** (the generic API origin — Partner is specifically the configured-Connection origin).

**External Reference**:
The *partner's own* identifier for a case — a peer CSIRT's ticket ID, or a vendor's advisory ID like `FG-IR-24-xxx` — carried in the email **subject** and used to **match** a later inbound email onto the **Incident** it already created. It is the partner-intake analogue of the phishing flow's `(sender_address, subject_normalised)` dedup: the join key that makes the **bi-directional** loop work, since updates from either side quote the same External Reference.
_Avoid_: **display_id** (the platform's *own* incident identifier; the External Reference is the *partner's*), **Correlation Key** (an ECS **Entity** a Correlation Rule binds on — an External Reference is an opaque partner string, not an entity).

**Vendor Advisory**:
The **Subject** carried by an **Incident** created from a **Vendor Connection**'s supplier bulletin (Fortinet, Palo Alto, …) — an *"assess whether we are affected"* case, not a confirmed detection. A human triages it: closing **no_impact** / **informational** when the fleet is unaffected, or escalating (by hand, in v1) when it is. v1 does **no** automatic affectedness check and **no** fan-out to affected customer orgs — both are deferred to a later phase (they need a product/version inventory to match against).
_Avoid_: **Vulnerability** (`source_kind = vulnerability` is a scanner-detected CVE on a specific **Asset**; a Vendor Advisory is a broadcast bulletin awaiting a human affectedness assessment).

**Intake Inbox**:
A staff-only dead-letter surface holding inbound emails that reached the SOC mailbox but **no handler accepted** — an unknown sender (matching no **Connection**), a Connection sender that **failed DKIM/SPF verification**, or an otherwise un-routable message (it also catches misfired phishing forwards). Each row carries bounded metadata (sender, subject, received-at, drop-reason, truncated body); its primary action is **"Create Connection"**, pre-filling the sender so a new partner is onboarded **going forward**. It is broader than partner intake but is the single place to see what is hitting the mailbox and bouncing. v1 does **not** replay a held message through the pipeline (deferred — issue #667); rows are auto-purged after a configurable retention window.
_Avoid_: a queue of **Incident**s (an Intake Inbox row is precisely a message that did *not* become an Incident/Alert), phishing quarantine (it is a routing dead-letter, not a spam verdict).

## Relationships

- A **Correlation Rule** is either a **System Rule** (org = null) or an **Org Rule**; it has one or more **Legs** and exactly one **Correlation Key**.
- A tenant evaluates: **System Rules** it has not **Muted**, plus its own **Org Rules**.
- A **Correlation Rule** that triggers produces one **Incident** and links the contributing **Alerts** to it.
- An **Alert** belongs to at most one **Incident** (existing `Alert.incident` FK).

**Supersede**:
What a triggering Correlation Rule does to a simpler incident that already owns one of its contributing alerts: relink the alerts to the chain incident, then mark the simpler incident `duplicate_of` it and close it. Reversible; history preserved on the closed incident. See [ADR-0002](docs/adr/0002-correlation-rules-supersede-via-duplicate-of.md).
_Avoid_: Merge (reserve for the deferred A-deep record-level migration).

## Flagged ambiguities

- "Combination of alerts" was ambiguous between a single-alert classifier (conditions over one alert) and a multi-leg correlation across heterogeneous alerts. **Resolved:** a Correlation Rule is multi-leg correlation; a single-leg rule is the degenerate case that subsumes the old severity auto-promote.
- The hardcoded `route_alert` steps (severity auto-promote, asset threshold) overlap with Correlation Rules. **Resolved (v1):** both run side by side — fast-path stays synchronous, Correlation Rules run async and **Supersede** any simpler incident they overlap. Eventually the hardcoded steps may be reframed as seed rules (deferred).
- A **Scheduled Search Rule** reuses the **Leg** / **Correlation Key** / **Window** vocabulary, but its match semantics are **co-occurrence within the window** ("all legs present for the same key"), *not* strict ordering ("leg A *then* leg B"). **Resolved (v1):** co-occurrence only, matching the streaming engine. Strict sequence/ordering is a recognised future axis (may matter once rules run in production for a while) — deferred.
- Process data is reachable two ways in a **Hunt**: the existing per-agent *live pull* over the syscollector REST API, and the new fleet *value-search* over the **IT Hygiene Inventory** indices. **Resolved (v1):** keep both — different access patterns (deep-dive one suspected host *now* vs. "which hosts run X?" across the fleet, which the per-agent pull cannot answer). Software and services exist *only* in the inventory indices. Consolidating onto one source is a recognised later cleanup — deferred.
- An **ingress-route Exposure**'s auto-*suggestion* matches the route's `backend_host` against a **Host Asset**'s IP/name. For `backend_type=direct` the backend string is the host's real address, so matching is tractable. For `backend_type=netbird` the `backend_host` is a NetBird overlay IP that won't match the host's Wazuh-reported `ip_address`, so suggestion is **deferred**: a NetBird route still gets the *explicit* link (operator can set `backend_asset` by hand) and produces a normal ingress-route Exposure — only the automatic suggestion is stubbed with a placeholder until NetBird-aware resolution is implemented.
- "Alert", "Incident", "Severity", and "monitoring" collide with **operational observability** (Prometheus alerts, the platform being down, pod-crash severity, app performance metrics). **Resolved:** observability — *operating the platform* — is a **separate concern from the SOC domain this glossary describes** (*the security signals the platform processes for tenants*). Operational terms are deliberately **kept out of this glossary**; a Prometheus "alert" is never an **Alert**, an app outage is never an **Incident**, and ops metrics are **tenant-agnostic** (carry no org identity) precisely to keep the two worlds apart. See ADR-0019.
- A **Connection** maps **one sender address → one target Organization** (senders are unique across Connections, so routing is unambiguous). A national **CSIRT** that reports for *many* customer orgs from a *single* address therefore cannot be handled in v1: routing each such mail to the *right* customer needs **per-message org resolution** (parse the mail's domain/asset/IP to decide which tenant it concerns) — the **same deferred "heavy lifting"** as **Vendor Advisory** affectedness-checking and fan-out. **Resolved (v1):** one sender → one org (a per-customer CSIRT relationship or a vendor); multi-org-per-sender routing is deferred to the later phase that adds product/asset inventory matching.
- A **Vendor Advisory** rides in the **Infrastructure organisation**, which ADR-0017 introduced strictly for **Shared Infrastructure events** (perimeter firewall/proxy telemetry). **Resolved (v1):** reuse it as the "belongs to no one tenant" home rather than standing up a separate internal/SOC org — a deliberate widening of the Infrastructure organisation's role from *shared-perimeter events* to *any fleet-level, non-tenant SOC record*. A dedicated internal-org split is a recognised future option if the two uses diverge.
